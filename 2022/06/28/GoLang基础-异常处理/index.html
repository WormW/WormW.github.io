GoLang基础-异常处理
发布于：2022-06-28 22:29:23 标签：/ golang / 基础 / 访问：

  1. Error

    1. Errors.New返回的是内存地址，为了避免直接比较时出现值的相等
    2. 其他语言的异常还是处理机制，还是向上继续抛出
    3. golang的panic是真正的中断
    4. 不要滥用二义性
    5. 不要滥用panic 与recover
    6. 不需要区分error和exception的语义了【很好
    7. Error are values

  2. 异常类型

    1. 预定义的特定错误，sentinel error 哨兵错误
      1. 不灵活的错误处理策略
      2. 不依赖检查error.Error的输出，不要使用Error的值作为方法判定，只做日志
      3. 增加API的公共部分，要暴露错误
      4. 在两个包之间创建了依赖（错误之间的依赖）
      5. 结论：尽可能避免sentinel error
    2. Error types 自定义error
      1. os.PathError
      2. 需要使用switch 和类型断言(.(type))
      3. 会导致自己的异常变为public，和调用者产生强耦合
    3. Opaque errors 不透明错误
      1. 对于不关心细节的调用方直接返回error
      2. 对于需要细节的调用方，使用暴露函数进行类型转换判定返回细节
    4. Wrap Error
      1. 错误原则
        1. 日志记录与错误无关且对调试没有帮助的信息被视为噪音
        2. 错误要被日志记录
        3. 应用程序处理错误，保证100%完整性
        4. 之后不再报告当前错误
      2. 错误处理
        1. 和其他库协作时考虑errors.Wrap或者errors.Wrapf
        2. 直接返回错误，不要每个地方都打日志
        3. 程序的顶部或者goroutine顶部 %+v 记录日志
        4. errors.Cause获取root error 再和sentinel error
        5. 业务应该wrap，但是基础库不应该wrap
        6. 处理不了的错误带上足够的信息向上返回
        7. 已经处理过的错误（降级处理），不应该继续向上抛出

  3. 异常处理 handling error

    1. 结构体包装错误和主流程（减少err!=nil）
    2. fmt.Errorf二次包装error（sentinel errors或者type assertions（error types的类型断言）不兼容 ）
    3. 错误只处理一次

  4. Unwrap递归寻找最终的底层error

    1. errors.Is
    2. errors.As
    3. 也可以通过自定义实现Is判定

  5. fmt.Errorf %w 包装错误

  6. pkg errors的优势还是堆栈信息的保留
